# Decisions

## 不做自动交易和行情预测

系统目标是提高记录质量、仓位纪律和复盘质量。券商连接、自动下单和确定性预测会引入账户安全、错误执行、数据授权及不可逆损失风险，因此不在当前范围内。

## 使用 Streamlit + SQLite

项目是单人本地工具。Streamlit 可直接复用 Python OCR、Pandas 和 Plotly 能力；SQLite 零服务、可迁移、易备份，适合当前规模。

## 当前不使用 React 重写

核心能力是 OCR、SQLite、Pandas、风险规则和数据导入，而不是复杂前端交互。Streamlit 更适合快速验证和迭代。React 重写会同时引入后端 API、前端状态管理、构建链路、部署和接口兼容成本，现阶段收益不足。等功能、数据模型和实际工作流稳定，且确有多用户或正式发布需求时，再评估 React/Vue + FastAPI。

## OCR 是可选依赖，导入必须人工确认

OCR 可能缺失、初始化失败或误识别。基础页面必须保持可用，并提供普通 OCR/手动文本回退。任何截图解析结果都必须进入确认表，不能自动写入持仓。

## 结构化快照而不是同步 SQLite

SQLite 二进制文件不适合 Git 合并。跨设备只提交 `data/sync/portfolio_sync.json`；导入支持预览、合并和覆盖，写入前必须备份。快照可能包含真实数据，只允许放在私有仓库。

## 重复截图默认覆盖持仓

同平台下用代码或规范化名称识别同一持仓；同批次保留最后结果。覆盖仅更新持仓核心字段，不删除交易、计划和复盘历史。
