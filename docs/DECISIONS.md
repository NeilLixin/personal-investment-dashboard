# Decisions

## 不做自动交易

系统目标是提高记录质量和仓位纪律。自动下单会引入账户安全、错误执行、监管与不可逆损失风险，因此不接券商交易接口。

## 第一版不用行情 API

行情源存在授权、限频、代码映射和错误报价问题。MVP 先验证持仓管理、计划和复盘是否真正有用，数据手动更新或截图导入。

## 使用 Streamlit + SQLite

Streamlit 能快速提供本地中文交互、表格和 Plotly 图表；SQLite 零服务、可备份、适合单人长期使用。

## OCR 是可选依赖

OCR 运行环境较重且可能初始化失败。基础页面必须始终可用，手动粘贴文本是可靠兜底。

## 导入前必须人工确认

OCR 和金融截图解析都可能误识别名称、金额或正负号。任何数据进入持仓前必须让用户修改和选择重复项策略。
# v0.2.0 决策

- 不同步 SQLite 二进制文件；只提交 `data/sync/portfolio_sync.json` 结构化快照。
- 快照包含真实持仓数据，因此仅限私有仓库；私有仓库也不是绝对安全。
- merge 采用同 ID 更新、无 ID 新增；overwrite 清空核心表后导入，两者都先备份。
- 日报和洞察完全由本地确定性规则生成，不接 OpenAI API，不预测涨跌，不自动交易。
# 为什么当前不使用 React 重写

当前项目是个人本地投资工具，核心能力是 OCR、SQLite、Pandas 和本地风险计算。Streamlit 可以让这些
Python 能力保持在同一运行环境中，也更适合快速迭代。React 重写意味着新增前后端 API、状态管理、
构建链路和部署体系，现阶段投入明显大于收益。等功能与数据模型稳定、确实需要面向更多用户发布时，
再评估 React/Vue + FastAPI 的正式产品架构。
